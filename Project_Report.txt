ABSTRACT
================================================================================

The Vehicle Service Record Database is a full-stack web application designed to
manage and track vehicle service records, appointments, and support tickets for
both bikes and cars. The system provides role-based access for administrators
and customers. Administrators can add, edit, and delete vehicles, manage
service records (maintenance, inspections, repairs), handle appointments, and
resolve support tickets. Customers can search for their vehicles by brand,
model, and vehicle ID, view their service history, book appointments, and raise
support tickets.

The backend is built using Node.js with Express.js framework and Sequelize ORM,
connected to a Neon PostgreSQL cloud database. The frontend uses HTML5, CSS3,
and Vanilla JavaScript with asynchronous REST API calls. The database consists
of five normalized relational tables: users, vehicles, service_records,
appointments, and tickets, with proper foreign key relationships and constraints.

Key features include vehicle-type filtering (Bike/Car), cost tracking with
properly formatted currency display, role-based dashboards, and a responsive
beige and dark grey themed UI.

1.1 Introduction
----------------
The Vehicle Service Record Database project addresses the need for a centralized
digital system to manage vehicle maintenance and service history. Traditional
paper-based methods of tracking vehicle services are error-prone, inefficient,
and difficult to search. This project provides a web-based solution where
service centres can maintain comprehensive digital records of all vehicle
services, and customers can conveniently check their vehicle service status.

The application supports two types of vehicles — Bikes and Cars — with 45
vehicles pre-loaded (40 bikes from 8 major Indian brands and 5 cars from 3
brands). It implements role-based access control with Admin and Customer portals.

1.2 Motivation
--------------
- Manual record-keeping leads to data loss and retrieval difficulties
- Customers need real-time access to their vehicle service history
- Service centres require efficient tracking of appointments and support tickets
- Need for digital transformation in the Indian automotive service industry

1.3 Scope
---------
- Multi-vehicle support (Bikes and Cars)
- Role-based access (Admin and Customer portals)
- CRUD operations for vehicles, service records, appointments, and tickets
- Real-time dashboard with statistics
- Vehicle-type filtering across all sections
- Cloud-hosted PostgreSQL database (Neon)

1.4 Problem Statement
---------------------
Design and implement a relational database system for vehicle service record
management that supports multiple vehicle types, role-based user access, service
history tracking, appointment scheduling, and customer support ticketing, with a
web-based frontend for interaction.

1.5 Project Requirements
------------------------
Functional Requirements:
  - User authentication with role-based access (Admin/Customer)
  - Vehicle management (Add, Edit, Delete, Search)
  - Service record tracking (Maintenance, Inspection, Repair with cost, mileage,
    technician, spares details)
  - Appointment booking and status management
  - Ticket/Query system for customer support
  - Dashboard with statistics and quick overview

Non-Functional Requirements:
  - Responsive web design
  - Cloud-hosted database for availability
  - RESTful API architecture
  - Secure session management

Technology Stack:
  - Backend: Node.js, Express.js, Sequelize ORM
  - Database: PostgreSQL (Neon Cloud)
  - Frontend: HTML5, CSS3, Vanilla JavaScript
  - Tools: npm, dotenv, cors

1.6 Identification of Entity and Relationships
-----------------------------------------------

Entities Identified:
  1. User       — stores admin and customer credentials and roles
  2. Vehicle    — stores bike and car details (type, brand, model, vehicle_id)
  3. ServiceRecord — stores service history for each vehicle
  4. Appointment — stores service appointment bookings
  5. Ticket     — stores customer support queries/issues

Relationships:
  - Vehicle (1) ----< (M) ServiceRecord     [One vehicle has many service records]
  - Vehicle (1) ----< (M) Appointment       [One vehicle has many appointments]
  - Vehicle (1) ----< (M) Ticket            [One vehicle has many tickets]
  - User    (1) ----< (M) Appointment       [One user books many appointments]
  - User    (1) ----< (M) Ticket            [One user raises many tickets]

2.3 Creation of Database and Tables – DDL Commands
---------------------------------------------------

-- Drop tables (reverse dependency order)
DROP TABLE IF EXISTS tickets CASCADE;
DROP TABLE IF EXISTS appointments CASCADE;
DROP TABLE IF EXISTS service_records CASCADE;
DROP TABLE IF EXISTS vehicles CASCADE;
DROP TABLE IF EXISTS users CASCADE;

-- Create Users Table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(100) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,
  full_name VARCHAR(255) NOT NULL,
  role VARCHAR(50) DEFAULT 'customer'
);

-- Create Vehicles Table
CREATE TABLE vehicles (
  id SERIAL PRIMARY KEY,
  type VARCHAR(50) NOT NULL,
  brand VARCHAR(100) NOT NULL,
  model VARCHAR(100) NOT NULL,
  vehicle_id VARCHAR(100) UNIQUE NOT NULL
);

-- Create Service Records Table
CREATE TABLE service_records (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id),
  service_type VARCHAR(100),
  service_date DATE,
  service_interval VARCHAR(100),
  mileage INTEGER,
  next_service_date DATE,
  description TEXT,
  cost NUMERIC(10,2) DEFAULT 0,
  technician_name VARCHAR(255),
  spares_changed TEXT,
  notes TEXT
);

-- Create Appointments Table
CREATE TABLE appointments (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id),
  user_id INTEGER REFERENCES users(id),
  appointment_date DATE NOT NULL,
  notes TEXT,
  status VARCHAR(50) DEFAULT 'Pending',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Create Tickets Table
CREATE TABLE tickets (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id),
  user_id INTEGER REFERENCES users(id),
  issue TEXT NOT NULL,
  status VARCHAR(50) DEFAULT 'Open',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

3.1 Adding Constraints and Queries Based on Constraints
--------------------------------------------------------

Question 1: Add a CHECK constraint to ensure vehicle type is either 'Bike' or
'Car', and query all vehicles that violate this constraint.

SQL Statement:
  ALTER TABLE vehicles
  ADD CONSTRAINT chk_vehicle_type CHECK (type IN ('Bike', 'Car'));

  -- Query all bikes
  SELECT * FROM vehicles WHERE type = 'Bike' ORDER BY brand;

Output: Returns all 40 bike records ordered by brand name.

Question 2: Add a NOT NULL constraint on the issue column of tickets and
attempt to insert a ticket without an issue.

SQL Statement:
  ALTER TABLE tickets
  ALTER COLUMN issue SET NOT NULL;

  -- This will fail due to NOT NULL constraint:
  INSERT INTO tickets (vehicle_id, user_id, status) VALUES (1, 2, 'Open');

Output: ERROR: null value in column "issue" violates not-null constraint

Question 3: Add a UNIQUE constraint on username and attempt a duplicate insert.

SQL Statement:
  ALTER TABLE users
  ADD CONSTRAINT uq_username UNIQUE (username);

  -- This will fail:
  INSERT INTO users (username, password, full_name, role)
  VALUES ('admin', 'test123', 'Test User', 'customer');

Output: ERROR: duplicate key value violates unique constraint "uq_username"

Question 2: Find the total cost, average cost, and count of service records
per service type.

SQL Statement:
  SELECT service_type,
         COUNT(*) AS total_records,
         SUM(cost) AS total_cost,
         ROUND(AVG(cost), 2) AS avg_cost,
         MAX(cost) AS max_cost
  FROM service_records
  GROUP BY service_type
  ORDER BY total_cost DESC;

Output: Returns aggregated service statistics grouped by Maintenance, Repair,
and Inspection types.

Question 3: Count the number of open tickets per user.

  GROUP BY u.full_name

3.4 Complex Queries Based on Subqueries
-----------------------------------------

Question 1: Find all vehicles that have no service records.

SQL Statement:
  SELECT v.vehicle_id, v.brand, v.model
  FROM vehicles v
  WHERE v.id NOT IN (
    SELECT DISTINCT vehicle_id FROM service_records
  );

Output: Returns all 45 vehicles (since no service records are seeded).

Question 2: Find users who have raised more tickets than the average.

SQL Statement:
  SELECT u.full_name, COUNT(t.id) AS ticket_count
  FROM users u
  JOIN tickets t ON u.id = t.user_id
  GROUP BY u.full_name
  HAVING COUNT(t.id) > (
    SELECT AVG(cnt) FROM (
      SELECT COUNT(id) AS cnt FROM tickets GROUP BY user_id
    ) AS avg_tickets
  );

Output:
  +----------+--------------+
  | full_name| ticket_count |
  +----------+--------------+
  | John Doe | 2            |
  +----------+--------------+

Question 3: Find vehicles belonging to brands that have more than 5 models.

SQL Statement:
  SELECT vehicle_id, brand, model
  FROM vehicles
  WHERE brand IN (
    SELECT brand FROM vehicles GROUP BY brand HAVING COUNT(*) >= 5
  )
  ORDER BY brand, model;

Output: Returns vehicles from Royal Enfield (7), Bajaj (5), KTM (5), TVS (5),
Honda (5), Yamaha (5).

ANALYZING THE PITFALLS, IDENTIFYING THE DEPENDENCIES, AND APPLYING
NORMALIZATIONS
================================================================================

4.1 Analyse the Pitfalls in Relations
---------------------------------------
Potential pitfalls in the Vehicle Service Record Database:

1. Insertion Anomaly: Cannot add a service record without first creating a
   vehicle. This is resolved by having separate vehicle and service_records
   tables with foreign keys.

2. Deletion Anomaly: Deleting a vehicle could cascade and lose all service
   history. The CASCADE constraint ensures referential integrity.

3. Update Anomaly: If vehicle brand/model were stored in service_records
   directly, updating a vehicle's brand would require updating all associated
   records. Normalization eliminates this.

4. Redundancy: Without normalization, customer name would be repeated across
   every ticket and appointment row. The users table eliminates this.


4.2 First Normal Form (1NF)
-----------------------------

4.2.1 Identify Dependency:
All tables have atomic values in each cell. No repeating groups exist.
Every table has a primary key (id).

4.2.2 Apply Normalization to 1NF:
The database is already in 1NF:
- Each column contains only atomic (indivisible) values
- Each row is unique (identified by primary key)
- No multi-valued attributes exist
- No repeating groups

Before 1NF (hypothetical unnormalized):
+----+--------+----------------------------------+
| id | brand  | services                         |
+----+--------+----------------------------------+
| 1  | Royal  | Oil Change, Brake Check, Filter  |
+----+--------+----------------------------------+

After 1NF:
+----+--------+
| id | brand  |
+----+--------+
| 1  | Royal  |
+----+--------+

+----+------------+--------------+
| id | vehicle_id | service_type |
+----+------------+--------------+
| 1  | 1          | Oil Change   |
| 2  | 1          | Brake Check  |
| 3  | 1          | Filter       |
+----+------------+--------------+


4.3 Second Normal Form (2NF)
------------------------------

4.3.1 Identify Dependency:
All non-key attributes are fully functionally dependent on the entire
primary key. No partial dependencies exist since all tables use single-
column surrogate keys (id).

4.3.2 Apply Normalization to 2NF:
The database is already in 2NF because:
- All tables use single-column primary keys (no composite keys)
- Therefore, no partial dependencies can exist
- Every non-key attribute depends on the full primary key

Example: In service_records table, all attributes (service_type,
service_date, cost, etc.) depend entirely on the primary key (id).


4.4 Third Normal Form (3NF)
------------------------------

4.4.1 Identify Dependency:
No transitive dependencies exist. Non-key attributes do not depend on
other non-key attributes.

4.4.2 Apply Normalization to 3NF:
The database is already in 3NF:
- In vehicles table: brand, model, type, vehicle_id all depend directly
  on id (PK), not on each other
- In tickets table: issue, status, created_at depend on id (PK), not on
  vehicle_id or user_id (which are FKs, not non-key determinants)

Hypothetical violation example (before 3NF):
+----+---------+-----------+-------------------+
| id | user_id | username  | full_name         |
+----+---------+-----------+-------------------+
(full_name depends on username, not on ticket id — transitive dependency)

After 3NF: User information is separated into the users table,
referenced by foreign key user_id.


4.5 BCNF (Boyce-Codd Normal Form)
-----------------------------------

4.5.1 Identify Dependency:
Every determinant is a candidate key. The only functional dependencies
are from primary keys and candidate keys (username UNIQUE in users,
vehicle_id UNIQUE in vehicles).

4.5.2 Apply Normalization to BCNF:
The database satisfies BCNF:
- users: id → all attributes; username → all attributes (candidate key)
- vehicles: id → all attributes; vehicle_id → all attributes (candidate key)
- Both determinants are candidate keys, satisfying BCNF


4.6 Fourth Normal Form (4NF)
------------------------------

4.6.1 Identify Dependency:
No multi-valued dependencies exist. Each fact is stored independently.

4.6.2 Apply Normalization to 4NF:
The database is in 4NF:
- A vehicle's tickets and appointments are stored in separate tables
- No single table contains two or more independent multi-valued facts
- Example: Vehicle → Tickets and Vehicle → Appointments are stored in
  separate tables (tickets, appointments), not combined


4.7 Fifth Normal Form (5NF)
------------------------------

4.7.1 Identify Dependency:
No join dependencies exist that aren't implied by candidate keys.

4.7.2 Apply Normalization to 5NF:
The database is in 5NF:
- The appointments table naturally combines vehicle_id, user_id, and
  appointment_date — this is a single fact, not decomposable further
  without loss
- No table can be further decomposed into smaller tables and then
  recombined via joins without information loss

5.1 Introduction to Transactions
-----------------------------------

A transaction is a sequence of one or more SQL operations treated as a single
logical unit of work. Transactions ensure data integrity and consistency in
the database.

5.1.1 Properties (ACID):
  A - Atomicity:    All operations complete or none do
  C - Consistency:  Database moves from one valid state to another
  I - Isolation:    Concurrent transactions don't interfere
  D - Durability:   Committed changes persist permanently

5.1.2 States:
  Active      → Transaction is executing operations
  Partially   → Final statement executed, awaiting commit
  Committed   → Changes are permanently saved
  Failed      → Error detected, cannot proceed
  Aborted     → Transaction rolled back to prior state


5.2 Transaction Control Language (TCL)
----------------------------------------

5.2.1 SAVEPOINT:
  Creates a point within a transaction to which you can roll back.
  Syntax: SAVEPOINT savepoint_name;

5.2.2 COMMIT:
  Permanently saves all changes made during the transaction.
  Syntax: COMMIT;

5.2.3 ROLLBACK:
  Reverts changes to the beginning of the transaction or to a savepoint.
  Syntax: ROLLBACK; or ROLLBACK TO savepoint_name;


5.3 Transactions for Vehicle Service Record Database
------------------------------------------------------

5.3.1 Transaction 1: Book Appointment with Validation

  BEGIN;

  -- Step 1: Insert a new appointment
  INSERT INTO appointments (vehicle_id, user_id, appointment_date, notes, status)
  VALUES (1, 2, '2026-03-01', 'Full bike service', 'Pending');

  SAVEPOINT after_appointment;

  -- Step 2: Try to insert a duplicate (error scenario)
  INSERT INTO appointments (vehicle_id, user_id, appointment_date, notes, status)
  VALUES (1, 2, '2026-03-01', 'Duplicate entry', 'Pending');

  -- Step 3: Rollback to savepoint (undo duplicate)
  ROLLBACK TO after_appointment;

  -- Step 4: Insert a valid second appointment
  INSERT INTO appointments (vehicle_id, user_id, appointment_date, notes, status)
  VALUES (8, 3, '2026-03-05', 'Chain lubrication', 'Pending');

  -- Step 5: Commit all valid changes
  COMMIT;


5.3.2 Transaction 2: Raise Ticket and Update Vehicle Status

  BEGIN;

  -- Step 1: Raise a new ticket
  INSERT INTO tickets (vehicle_id, user_id, issue, status)
  VALUES (5, 2, 'Front suspension feels loose', 'Open');

  SAVEPOINT after_ticket;

  -- Step 2: Try raising a ticket with invalid vehicle_id
  INSERT INTO tickets (vehicle_id, user_id, issue, status)
  VALUES (999, 2, 'Invalid vehicle', 'Open');

  -- Step 3: Rollback invalid insert
  ROLLBACK TO after_ticket;

  -- Step 4: Update existing ticket status
  UPDATE tickets SET status = 'In Progress' WHERE id = 1;

  COMMIT;


5.3.3 Transaction 3: Add Service Record with Cost Validation

  BEGIN;

  -- Step 1: Add a service record
  INSERT INTO service_records (vehicle_id, service_type, service_date,
    description, cost, technician_name, spares_changed)
  VALUES (1, 'Maintenance', '2026-02-10',
    'Regular service - oil change and filter', 5000, 'Rajesh Kumar',
    'Engine Oil, Oil Filter, Air Filter');

  SAVEPOINT after_service;

  -- Step 2: Insert a record with negative cost (invalid)
  INSERT INTO service_records (vehicle_id, service_type, service_date,
    description, cost) VALUES (1, 'Repair', '2026-02-10', 'Test', -500);

  -- Step 3: Rollback invalid record
  ROLLBACK TO after_service;

  -- Step 4: Insert valid second service
  INSERT INTO service_records (vehicle_id, service_type, service_date,
    description, cost, technician_name)
  VALUES (1, 'Inspection', '2026-02-10', 'General inspection', 500,
    'Amit Sharma');

  COMMIT;


5.3.4 Transaction 4: Bulk Update Ticket Statuses

  BEGIN;

  -- Step 1: Close all "Open" tickets
  UPDATE tickets SET status = 'Closed' WHERE status = 'Open';

  SAVEPOINT after_close_all;

  -- Step 2: Reopen specific ticket (changed mind)
  UPDATE tickets SET status = 'Open' WHERE id = 2;

  -- Step 3: Commit with selective reopening
  COMMIT;


5.3.5 Transaction 5: Delete Vehicle with Cascading Cleanup

  BEGIN;

  -- Step 1: Delete all tickets for vehicle 45
  DELETE FROM tickets WHERE vehicle_id = 45;

  SAVEPOINT after_tickets_delete;

  -- Step 2: Delete all appointments for vehicle 45
  DELETE FROM appointments WHERE vehicle_id = 45;

  SAVEPOINT after_appointments_delete;

  -- Step 3: Delete all service records for vehicle 45
  DELETE FROM service_records WHERE vehicle_id = 45;

  -- Step 4: Delete the vehicle itself
  DELETE FROM vehicles WHERE id = 45;

  -- Step 5: Commit entire cascade
  COMMIT;


5.3 Concurrency Control
-------------------------

5.3.1 Concurrency Control Algorithms:
Concurrency control ensures that multiple transactions can execute
simultaneously without compromising data integrity. PostgreSQL uses
Multi-Version Concurrency Control (MVCC) as its primary mechanism.

5.3.1 Locking Commands:

a. Row-Level Locking – SELECT ... FOR UPDATE
   Locks specific rows to prevent concurrent modification.

b. Table-Level Locking – LOCK TABLE
   Locks the entire table for exclusive access.

Lock Modes:
+---------------------+----------------------------------------------------------+
| Lock Mode           | Description                                              |
+---------------------+----------------------------------------------------------+
| ROW SHARE           | Allows concurrent access; prevents exclusive table locks |
| ROW EXCLUSIVE       | Prevents share mode locks. Default for DML operations    |
| SHARE               | Allows queries but not updates or deletes                |
| SHARE ROW EXCLUSIVE | More restrictive than SHARE                              |
| EXCLUSIVE           | Prevents all other access — full table lock               |
+---------------------+----------------------------------------------------------+

c. COMMIT – Release All Locks
   After COMMIT, all locks held by the transaction are released.

d. ROLLBACK – Undo Changes & Release Locks
   After ROLLBACK, changes are undone and all locks are released.

5.3.2 Example: Concurrent Appointment Booking

  -- Session 1: Lock the appointment slot to prevent double-booking
  BEGIN;
  SELECT * FROM appointments
  WHERE vehicle_id = 1 AND appointment_date = '2026-02-15'
  FOR UPDATE;

  -- Session 1 holds the lock while processing...
  UPDATE appointments SET status = 'Confirmed'
  WHERE vehicle_id = 1 AND appointment_date = '2026-02-15';

  COMMIT;
  -- Lock is released after commit

  -- Session 2 (concurrent): Would wait until Session 1 commits
  BEGIN;
  SELECT * FROM appointments
  WHERE vehicle_id = 1 AND appointment_date = '2026-02-15'
  FOR UPDATE;
  -- This blocks until Session 1 commits or rolls back
  COMMIT;
