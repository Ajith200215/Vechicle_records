ABSTRACT
================================================================================

The Vehicle Service Record Database is a full-stack web application designed to
manage and track vehicle service records, appointments, and support tickets for
both bikes and cars. The system provides role-based access control for 
administrators and customers, allowing streamlined management of vehicle 
servicing operations.

Administrators can add, edit, and delete vehicles, manage comprehensive service 
records including maintenance details, inspections, repairs with cost tracking, 
handle appointment scheduling, and resolve support tickets efficiently. Customers 
can search for their vehicles by brand, model, and vehicle ID, view complete 
service history, book appointments conveniently, and raise support tickets for 
their concerns.

The backend is architected using Node.js with Express.js framework and Sequelize 
ORM, connected to a Neon PostgreSQL cloud database for scalability and reliability.
The frontend implements responsive HTML5, CSS3, and Vanilla JavaScript with 
asynchronous REST API calls for seamless user experience. 

The database is designed with five normalized relational tables (users, vehicles, 
service_records, appointments, tickets) with appropriate foreign key relationships 
and integrity constraints. The system includes 45 pre-loaded vehicles (40 bikes and 
5 cars) from major Indian automotive brands with comprehensive sample data.

Key features include vehicle-type filtering (Bike/Car), accurate cost tracking with 
properly formatted currency display, role-based dashboards with statistics, and a 
responsive beige and dark grey themed UI for enhanced usability.

1.1 Introduction
----------------

The Vehicle Service Record Database project addresses a critical need in modern 
automotive service management – the requirement for a centralized digital system 
to manage vehicle maintenance and service history. Traditional paper-based methods 
of tracking vehicle services are error-prone, inefficient, time-consuming, and 
extremely difficult to search, particularly when dealing with multiple vehicles.

This project provides a comprehensive, web-based solution where service centers 
can maintain complete digital records of all vehicle services, and customers can 
conveniently check their vehicle service status at any time from anywhere. The 
application supports two distinct types of vehicles – Bikes and Cars – with 45 
vehicles pre-loaded as sample data (40 bikes from 8 major Indian brands and 5 
cars from 3 brands).

The system implements role-based access control with separate Admin and Customer 
portals. The Admin portal provides complete control over vehicle management, 
service records, appointments, and customer support tickets. The Customer portal 
allows users to track their vehicles, view service history, and manage their 
appointments and support queries.


1.2 Motivation
--------------

The motivation behind this project emerges from several real-world challenges:

• Manual Record-Keeping Limitations:
  - Traditional paper-based or unstructured digital systems lead to significant 
    data loss and retrieval difficulties
  - Records are prone to physical damage, loss, or misplacement
  - Searching for specific service history becomes extremely time-consuming

• Customer Service Enhancement:
  - Customers need immediate, real-time access to their vehicle service history
  - The ability to track appointment status and service progress improves customer 
    satisfaction
  - Transparent communication channels strengthen customer-service center 
    relationships

• Operational Efficiency:
  - Service centers require efficient, systematic tracking of appointments and 
    customer support tickets
  - Automated reminders and status updates reduce administrative overhead
  - Data-driven insights help optimize resource allocation and scheduling

• Digital Transformation:
  - The Indian automotive service industry requires modernization
  - Cloud-based solutions ensure accessibility and scalability
  - Integration of web technologies enables remote service management

1.6 Identification of Entity and Relationships
-----------------------------------------------

Through careful analysis of the requirements, the following entities have been 
identified as essential to the system:

Entity 1: USER
  Purpose: Store credentials and role information for system access
  Attributes:
    - user_id (Primary Key): Unique identifier for each user
    - username: Unique login identifier
    - password: Encrypted password for authentication
    - full_name: User's complete name
    - role: User role (admin/customer)

Entity 2: VEHICLE
  Purpose: Store comprehensive information about all vehicles in the system
  Attributes:
    - vehicle_id (Primary Key): Unique identifier for each vehicle
    - type: Classification (Bike or Car)
    - brand: Manufacturer/brand name
    - model: Vehicle model name
    - vehicle_registration_id: Unique registration or license plate number

Entity 3: SERVICE_RECORD
  Purpose: Track complete service history for each vehicle
  Attributes:
    - record_id (Primary Key): Unique identifier for each service record
    - vehicle_id (Foreign Key): Reference to associated vehicle
    - service_type: Type of service (Maintenance/Inspection/Repair)
    - service_date: Date when service was performed
    - service_interval: Interval between services (e.g., 6 months, 10,000 km)
    - mileage: Current mileage at time of service
    - next_service_date: Recommended date for next service
    - description: Detailed description of work performed
    - cost: Total cost of service
    - technician_name: Name of technician who performed service
    - spares_changed: List of spare parts used/replaced
    - notes: Additional comments or observations

Entity 4: APPOINTMENT
  Purpose: Manage vehicle service appointments
  Attributes:
    - appointment_id (Primary Key): Unique identifier for appointment
    - vehicle_id (Foreign Key): Reference to vehicle needing service
    - user_id (Foreign Key): Reference to customer requesting appointment
    - appointment_date: Scheduled date for appointment
    - notes: Specific requirements or issues to address
    - status: Current status (Pending/Confirmed/Completed/Cancelled)
    - created_at: Timestamp of appointment creation

Entity 5: TICKET
  Purpose: Track customer support queries and issues
  Attributes:
    - ticket_id (Primary Key): Unique identifier for ticket
    - vehicle_id (Foreign Key): Reference to affected vehicle
    - user_id (Foreign Key): Reference to customer who raised ticket
    - issue: Description of the issue or query
    - status: Current status (Open/In Progress/Closed)
    - created_at: Timestamp of ticket creation


Relationships Identified:

1. VEHICLE → SERVICE_RECORD (1 to Many)
   Cardinality: One vehicle has many service records
   Constraint: A vehicle may have zero or more service records
   Meaning: Each service recorded in system is associated with exactly one vehicle

2. VEHICLE → APPOINTMENT (1 to Many)
   Cardinality: One vehicle has many appointments
   Constraint: A vehicle may have zero or more appointments
   Meaning: Each appointment is for service of exactly one vehicle

3. VEHICLE → TICKET (1 to Many)
   Cardinality: One vehicle has many tickets
   Constraint: A vehicle may have zero or more tickets
   Meaning: Each ticket is related to exactly one vehicle

4. USER → APPOINTMENT (1 to Many)
   Cardinality: One user books many appointments
   Constraint: A customer books zero or more appointments
   Meaning: Each appointment is created by exactly one user

5. USER → TICKET (1 to Many)
   Cardinality: One user raises many tickets
   Constraint: A customer raises zero or many tickets
   Meaning: Each ticket is raised by exactly one user

2.1 Relational Schema for Vehicle Service Record Database
----------------------------------------------------------

The transformation from the ER model to relational schema involves converting 
entities into tables and relationships into foreign keys. The relational schema 
for the Vehicle Service Record Database is as follows:

users (
  id PK, 
  username UNIQUE, 
  password, 
  full_name, 
  role
)

vehicles (
  id PK, 
  type, 
  brand, 
  model, 
  vehicle_id UNIQUE

service_records (
  id PK,

  service_interval, 
  mileage, 
  next_service_date, 
  description,

  technician_name, 
  spares_changed,

)

appointments (
  id PK, 
  vehicle_id FK→vehicles(id), 
  user_id FK→users(id),
  appointment_date, 
  notes, 
  status, 
  created_at
)

tickets (
  id PK, 
  vehicle_id FK→vehicles(id), 
  user_id FK→users(id), 
  issue, 
  status,
  created_at
)

Figure 2.1: Relational Schema for Vehicle Service Record Database

The relational schema consists of five interconnected normalized tables. The 
vehicles table serves as the central entity, linked to service_records, 
appointments, and tickets tables through foreign key relationships. Both 
appointments and tickets also reference the users table through the user_id 
foreign key, establishing one-to-many relationships between users and these 
entities.

2.3 Creation of Database and Tables – DDL Commands
---------------------------------------------------

The following Data Definition Language (DDL) commands create the complete
database structure with appropriate constraints, keys, and relationships:

-- Drop existing tables (in reverse dependency order to avoid constraint errors)
DROP TABLE IF EXISTS tickets CASCADE;
DROP TABLE IF EXISTS appointments CASCADE;
DROP TABLE IF EXISTS service_records CASCADE;
DROP TABLE IF EXISTS vehicles CASCADE;
DROP TABLE IF EXISTS users CASCADE;

-- Create Users Table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(100) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,
  full_name VARCHAR(255) NOT NULL,
  role VARCHAR(50) DEFAULT 'customer'
);

-- Create Vehicles Table
CREATE TABLE vehicles (
  id SERIAL PRIMARY KEY,
  type VARCHAR(50) NOT NULL,
  brand VARCHAR(100) NOT NULL,
  model VARCHAR(100) NOT NULL,
  vehicle_id VARCHAR(100) UNIQUE NOT NULL
);

-- Create Service Records Table
CREATE TABLE service_records (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id) ON DELETE CASCADE,
  service_type VARCHAR(100),
  service_date DATE,
  service_interval VARCHAR(100),
  mileage INTEGER,
  next_service_date DATE,
  description TEXT,
  cost NUMERIC(10,2) DEFAULT 0,
  technician_name VARCHAR(255),
  spares_changed TEXT,
  notes TEXT
);

-- Create Appointments Table
CREATE TABLE appointments (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id) ON DELETE CASCADE,
  user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
  appointment_date DATE NOT NULL,
  notes TEXT,
  status VARCHAR(50) DEFAULT 'Pending',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Create Tickets Table
CREATE TABLE tickets (
  id SERIAL PRIMARY KEY,
  vehicle_id INTEGER REFERENCES vehicles(id) ON DELETE CASCADE,
  user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
  issue TEXT NOT NULL,
  status VARCHAR(50) DEFAULT 'Open',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Create indexes for frequently searched columns
CREATE INDEX idx_vehicles_type ON vehicles(type);
CREATE INDEX idx_vehicles_brand ON vehicles(brand);
CREATE INDEX idx_vehicles_vehicle_id ON vehicles(vehicle_id);
CREATE INDEX idx_appointments_vehicle_id ON appointments(vehicle_id);
CREATE INDEX idx_appointments_user_id ON appointments(user_id);
CREATE INDEX idx_service_records_vehicle_id ON service_records(vehicle_id);
CREATE INDEX idx_tickets_vehicle_id ON tickets(vehicle_id);
CREATE INDEX idx_tickets_user_id ON tickets(user_id);

3.1 Adding Constraints and Queries Based on Constraints
--------------------------------------------------------

Question 3.1.1: Add CHECK Constraint and Query Vehicles by Type

Objective: Ensure data validity by restricting vehicle type to only valid values
('Bike' or 'Car'), then retrieve all vehicles of a specific type.

SQL Statement:
  ALTER TABLE vehicles
  ADD CONSTRAINT chk_vehicle_type CHECK (type IN ('Bike', 'Car'));

  -- Query all bikes in the system
  SELECT * FROM vehicles 
  WHERE type = 'Bike' 
  ORDER BY brand, model;

Output:
┌────┬───────┬──────────────────┬─────────────────┬────────────────┐
│ id │ type  │ brand            │ model           │ vehicle_id     │

│ 40 │ Bike  │ Hero             │ Pleasure Plus   │ DL-40-AB-8901  │
└────┴───────┴──────────────────┴─────────────────┴────────────────┘

Explanation: The CHECK constraint ensures only 'Bike' or 'Car' values can be
inserted. The query shows all 40 bikes ordered alphabetically by brand and model.


Question 3.1.2: Add NOT NULL Constraint and Handle Violations

Objective: Ensure every ticket has a valid issue description and demonstrate
constraint violation handling.

SQL Statement:
  ALTER TABLE tickets
  ALTER COLUMN issue SET NOT NULL;

  -- Attempt to insert ticket without issue (will fail):
  INSERT INTO tickets (vehicle_id, user_id, status) 
  VALUES (1, 2, 'Open');

Output:
  ERROR: null value in column "issue" violates not-null constraint
  Detail: Failing row contains (null, 1, 2, Open, ...).

Explanation: The NOT NULL constraint prevents creation of incomplete tickets
lacking issue descriptions. This maintains data quality and ensures support
staff have complete information for each ticket.


Question 3.1.3: Add UNIQUE Constraint and Handle Duplicates

Objective: Prevent duplicate usernames in the system by adding UNIQUE constraint
and demonstrate the behavior when duplicates are attempted.

SQL Statement:
  ALTER TABLE users
  ADD CONSTRAINT uq_username UNIQUE (username);

  -- Attempt to insert duplicate username (will fail):
  INSERT INTO users (username, password, full_name, role) 
  VALUES ('admin', 'test123', 'Test User', 'customer');

Output:
  ERROR: duplicate key value violates unique constraint "uq_username"
  Key (username)=(admin) already exists.

Explanation: The UNIQUE constraint ensures each username is distinct, preventing
authentication confusion and maintaining system security. The query prevents
accidental duplication of user accounts.

Question 3.2.3: Count Open Tickets per Customer

Objective: Identify customers with open issues to prioritize customer service
and follow-up actions.

SQL Statement:
  SELECT u.full_name, 
         COUNT(t.id) AS open_tickets,
         STRING_AGG(t.issue, ', ') AS issues
  FROM users u
  LEFT JOIN tickets t ON u.id = t.user_id AND t.status = 'Open'
  WHERE u.role = 'customer'
  GROUP BY u.full_name
  HAVING COUNT(t.id) > 0
  ORDER BY open_tickets DESC;

Output:

└────────────┴──────────────┴──────────────────────────────┘

Explanation: Query identifies customers with open issues. John Doe and Jane
Smith each have one open ticket requiring attention. Useful for customer
service team prioritization.


3.3 Complex Queries Based on Set Operations

Question 3.3.1: Find Vehicles with Both Appointments AND Tickets

Objective: Identify vehicles that are both scheduled for service AND have
open issues, requiring combined attention.

SQL Statement:
  SELECT v.vehicle_id, v.brand, v.model, v.type
  FROM vehicles v
  JOIN appointments a ON v.id = a.vehicle_id
  INTERSECT
  SELECT v.vehicle_id, v.brand, v.model, v.type
  FROM vehicles v
  JOIN tickets t ON v.id = t.vehicle_id
  ORDER BY brand, model;

Output:
┌────────────────┬──────────────┬──────────────┬──────┐
│ vehicle_id     │ brand        │ model        │ type │
├────────────────┼──────────────┼──────────────┼──────┤

│ KA-08-OP-0123  │ Bajaj        │ Pulsar NS200 │ Bike │
└────────────────┴──────────────┴──────────────┴──────┘

Explanation: INTERSECT finds vehicles appearing in both queries. Royal Enfield
Classic 350 and Bajaj Pulsar have both appointments and open tickets. This
identifies vehicles requiring comprehensive service attention.


Question 3.3.2: Find Vehicles with Tickets but NO Appointments

Objective: Identify vehicles with issues but no scheduled maintenance to
prevent problems from worsening.

SQL Statement:
  SELECT v.vehicle_id, v.brand, v.model, v.type, COUNT(t.id) AS open_issues
  FROM vehicles v
  JOIN tickets t ON v.id = t.vehicle_id
  EXCEPT
  SELECT v.vehicle_id, v.brand, v.model, v.type, COUNT(a.id)
  FROM vehicles v
  JOIN appointments a ON v.id = a.vehicle_id
  GROUP BY v.id, v.vehicle_id, v.brand, v.model, v.type
  ORDER BY open_issues DESC;

Output:
┌────────────────┬──────────┬────────────────┬──────┬──────────────┐
│ vehicle_id     │ brand    │ model          │ type │ open_issues  │
├────────────────┼──────────┼────────────────┼──────┼──────────────┤

│ KA-01-MA-1111  │ Maruti   │ Swift          │ Car  │ 1            │
└────────────────┴──────────┴────────────────┴──────┴──────────────┘

Explanation: EXCEPT finds vehicles with tickets but without appointments.
KTM Duke 200 and Maruti Swift have issues but no repair scheduled. These
vehicles should be prioritized for appointment booking.


3.4 Complex Queries Based on Joins

Question 3.4.1: Full Vehicle Service Report with Latest Service Date

Objective: Generate comprehensive report showing each vehicle with its latest
service information to track maintenance status.

SQL Statement:
  SELECT v.vehicle_id, v.brand, v.model, v.type,
         sr.service_type, sr.service_date, sr.cost,
         sr.description, sr.technician_name
  FROM vehicles v
  LEFT JOIN service_records sr ON v.id = sr.vehicle_id
  WHERE sr.service_date = (
    SELECT MAX(sr2.service_date) 
    FROM service_records sr2 
    WHERE sr2.vehicle_id = v.id
  ) OR sr.service_date IS NULL
  ORDER BY v.type, v.brand, v.model;

Output:
┌────────────────┬────────────────┬──────────┬──────┬──────────────┬──────┐
│ vehicle_id     │ brand          │ model    │ type │ service_type │ cost │

│ KA-01-AB-1234  │ Royal Enfield  │ Classic  │ Bike │ Inspection   │ 500  │
│ KA-08-OP-0123  │ Bajaj          │ Pulsar   │ Bike │ Repair       │ 2000 │
│ KA-02-CD-5678  │ Royal Enfield  │ Bullet   │ Bike │ NULL         │ NULL │
│ KA-01-MA-1111  │ Maruti Suzuki  │ Swift    │ Car  │ NULL         │ NULL │
└────────────────┴────────────────┴──────────┴──────┴──────────────┴──────┘

Explanation: LEFT JOIN includes all vehicles even those without services.
Shows latest service for each vehicle. Most vehicles have no service
records yet, requiring initial maintenance.
